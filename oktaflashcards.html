Three types of users explained

What are the most-common types of users?

Okta Sourced, Directory-Sourced, App-Sourced

What is an Okta-Sourced user?

Account that is maintained within Okta; the attributes modified within Okta’s Identity cloud. Their authentication goes against Okta.

What is an Directory-Sourced user?

Account is created from an directory integration (on-prem AD, LDAP). Their attributes are sourced from the directory service. They authenticate against the directory, if delegated authentication is enabled (default setting).

What is an Application-Sourced user?

The source of truth for the account is an application, like a HR system. Their authentication goes against Okta or directory.

How many default base attributes are in the Okta User Profile and which are required by default?

31 default base attributes. Default required attributes are; Username, Primary Email, First Name, Last Name. (First Name and Last Name can be made Optional).

What are the different User Statuses in Okta?

Staged, Pending User Action, Active, Deactivated, Password Reset, Password Expired, Locked Out, and Suspended

What does the Staged user status mean?

The user account is created, but they have not been activated yet. There may be a pending admin action required.

Does not count toward an available Okta license.

What does the Pending User Action status mean?

The user account is created and activated, but the user has not completed a required action, such as using the welcome email to setup a password.

In the API or Reporting, this status is showed as “Provisioned”.

Does not count toward an available Okta license.

What does the Active user status mean?

The user account is active and they can access all assigned applications.

What does the Deactivated user status mean?

User account is inactive. Removes all applications, and the password is permanently deleted for the account. Admin can’t assign any applications to the account.

In Reporting and API, shown as “Deprovisioned”.

Does not count toward an available Okta license.

What does the Password Reset user status mean?

The account have a recovery status. Users can request a password reset or an admin can initiate one on their behalf. User can not sign into their account.

In Reporting & API, shown as “Recovery”.

Uses an active Okta license.

What does the Password Expired user status mean?

The password has expired and needs to be updated before the user can get access to applications. The user can not sign into their account.

Uses an active Okta license.

What does the Locked Out user status mean?

The account has exceed the number of allowed login attempts defined in the login policy. The user can not sign into their account until unlocked by an admin.

Uses an active Okta license.

What does the Suspended user status mean?

User cannot access applications or dashboard, but application assignments are unaffected and the user profile can be updated.

Uses an active Okta license.

Create Okta-Sourced users individually

What are 3 different ways Okta-sourced users can be created?

Individually via the admin console, via the API, or in-bulk via a CSV import

While creating Okta-sourced users individually via the admin console, what conditions do you need to use to create them in a Staged status?

In the user creation wizard (after selecting Create user), set the Activation setting to “Activated Later”.

While creating Okta-sourced users individually via the admin console, what conditions do you need to use to create them in a “Pending User Action” status?

In the user creation wizard (after selecting Create user), set the Activation setting to “Activated Now”, but leave the “I will set password” box unchecked.

While creating Okta-sourced users individually via the admin console, what conditions do you need to use to create them in an “Active” status?

In the user creation wizard (after selecting Create user), set the Activation setting to “Activated Now” and check the box “I will set password”.

You can optionally select “User must change password on first login” as need be.

When creating new Okta accounts, what format must the username be in?

Username must be in an email format, but doesn’t need to be a function email address

Where are activation emails sent?

Activation emails are sent to the primary and secondary email addresses for the account.

Password policy requirements are shown while users create passwords for Okta

Create Okta-Sourced Users in Bulk

What is the size limit of a In-build user csv file?

Up to 10,000 records OR up to 10 MB of data, but CSV imports are not designed for large-scale user imports

True or False; Importing users in bulk only creates new “Staged” user accounts in Okta

False; importing a CSV can update existing users by a match via the unique username of each user. You can also specify to activate new accounts during the import process.

True or False: Okta validates the CSV file format before importing users in bulk

True, by selecting “Upload CSV” during the import process, it parses and validates the CSV file.

Managed Privileged Accounts

Why do you need to have various types of admin roles?

To limit access to privilege users, or divide what actions admins can do based on their role. Okta provides both Standard Admin Roles and you can create custom admin roles to answer this requirement

What type of users can have admin roles assigned?

All types (Okta-sourced, directory-sourced, and application-sourced) can, but the Best practice is that Okta-sourced users have admin roles

What are the main predefined admin roles?

Super Admin, Org Admin, Group Admin, Group Membership Admin, Application Admin, Help Desk Admin, Read-only Admin, Report Admin, Mobile Admin, and API Access Management Admin

Describe what the Super Admin role provides

Full access to perform all admin tasks and permission set in Okta. It is the only role that can assign admin privileges. Highest level

(was called Super Organization Admin in the past)

Describe what the Organization Admin role provides

Can perform most org-wide settings. Can perform all management tasks for users and groups. cannot perform any application management tasks (assigning them)

Describe what the Application Admin role provides

Can manage profile information, can view users and groups, but not modify either. Can manage info on apps to which they are assigned access.

Describe what the Group admin role provides

Can create users, deactivate users, and reset passwords. Can be used to restrict task to select group/groups of Okta users. Cannot perform group creation and any app management tasks.

Describe what the Help Desk admin role provides

Can view users, can reset password and MFA, can clear user session, and can unlock users

Describe what the Read-only admin role provides

Can view and run reports. Can view users, groups and apps, but cannot modify any settings. Can view, but not modify org settings.

Describe what the Mobile Admin role provides

can perform actions related to mobile policies, sign-on policies, mobile devices, and Okta Mobile.

Create Custom Admin Roles

What are the 3 components to custom admin roles?

Recourses - The groups and applications the role is constrained to (what they can access)

Roles - The permission sets that the Admins have (what actions they can take)

Admins - The groups or individual accounts

How do you update the notification settings for admins?

Go to Security > Administrators > Roles, then select the Edit action for the role you would like to update notification settings for.

True or False: The resource set is the permissions an admin account is constrained to.

False: The Roles are the permission set the admin account has.

What report shows admin roles & resource sets that are assigned in your organization?

The Admin Role Assignments report
